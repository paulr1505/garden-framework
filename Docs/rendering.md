# Rendering

Most rendering is automatic: attach a `MeshComponent`, give it a transform, place a camera, and the deferred renderer draws it. This page covers what game code can and should *do* with the renderer, not the renderer's internals (those are in `../Documentation/Features.md`).

## Backends

| Backend | Platforms |
| :--- | :--- |
| Direct3D 12 | Windows (default) |
| Vulkan | Windows, Linux, macOS |
| Metal | macOS (default) |
| Headless | Servers |

The host picks a backend before your DLL loads, based on command-line flags or saved editor settings. By the time `gardenGameInit` runs, `g_services->render_api` is the active backend's `IRenderAPI*`. Game code is backend-agnostic — you do not branch on which backend is active.

## Drawing a mesh

```cpp
auto& reg = g_services->game_world->registry;
auto e = reg.create();

reg.emplace<TransformComponent>(e, 0.0f, 0.0f, 0.0f);

auto& m = reg.emplace<MeshComponent>(e);
m.m_mesh = AssetManager::get().loadMesh("assets/models/crate.glb");
```

The mesh is rendered each frame by the renderer's mesh pass. To hide a mesh, remove `MeshComponent` (or set its `m_mesh` to `nullptr` — the system skips null meshes).

## Cameras

The active rendering camera lives at `g_services->game_world->world_camera`. To follow the player:

```cpp
auto& cam = g_services->game_world->world_camera;
auto& tx  = reg.get<TransformComponent>(g_player_entity);
cam.position = tx.position + glm::vec3(0, 1.7f, 0);   // eye height
cam.yaw      = player_yaw;
cam.pitch    = player_pitch;
```

In the editor, the editor camera takes over outside of PIE — you don't have to do anything special.

## Materials

Garden meshes carry per-submesh materials loaded from glTF. To override:

- Edit the source `.gltf` / `.glb` and reimport (Content Browser ▸ right-click ▸ Reimport).
- Or attach a custom override via the mesh's material slots from C++ — see `Engine/src/Components/mesh.hpp`.

Material textures must be `.ctex` (compiled). The asset compiler converts source PNG/JPG when you reimport. Don't ship raw textures.

## LODs

`mesh` carries up to N LODs generated by meshoptimizer. The renderer picks an LOD per draw based on screen-space size. Configure thresholds with the **LOD Settings** panel in the editor — saved into the model's metadata.

## Lighting

The deferred pipeline supports:

- One **directional light** with cascaded shadow maps (4 cascades, PCF). Direction and color are set per-level via the *Level Settings* panel, or programmatically through the level system.
- **Skybox** — procedural atmosphere by default.
- Punctual lights (point / spot) are not yet first-class components in this branch — check `Engine/src/Graphics` for current state.

## Post-processing

The post-process chain (SSAO, shadow mask, tonemap, FXAA) is composed via `PostProcessGraphBuilder::Config`. For game code, the typical knobs are toggling SSAO on/off and adjusting exposure. These are surfaced through ConVars (e.g. `r_ssao`, `r_exposure` — confirm names in `Engine/src/Console`).

## Debug drawing

```cpp
#include "Debug/DebugDraw.hpp"

DebugDraw::get().drawLine(a, b, glm::vec3(1,0,0), 1.0f /* lifetime, seconds */);
DebugDraw::get().drawBox(center, half_extents, glm::vec3(0,1,0));
DebugDraw::get().drawSphere(center, radius, glm::vec3(1,1,0));
DebugDraw::get().drawCapsule(a, b, radius, glm::vec3(0,1,1));
```

Submitted shapes survive their lifetime then disappear. Use this for AI sight cones, gunfire traces, navigation hints during development.

## Custom shaders

Runtime shader sources live in `assets/shaders/slang/` and are compiled by `compile_shaders_slang.bat` / `.sh` to `assets/shaders/compiled/<backend>/`. To add a custom shader for game-specific rendering, you currently need to:

1. Add a `.slang` file under `assets/shaders/slang/`.
2. Run the compile script.
3. Drive the resulting pipeline through `IRenderAPI` directly.

This is rare for gameplay code — most games never write a shader and rely on glTF materials. If you do, the engine's existing shaders under `assets/shaders/slang/` are the reference.

## Frustum culling

Frustum culling is automatic, BVH-accelerated. You don't toggle it. Off-screen meshes are skipped without you asking.

## Performance pitfalls

- **Spawning many tiny meshes.** Each mesh is a draw call. Combine static scenery into fewer meshes in your DCC, or load instanced variants into a single `MeshComponent`.
- **Editing `TransformComponent` of a body without a rigid body.** Static colliders update lazily; if you move them, mark them with the appropriate flag (see `PhysicsSystem`).
- **Destroying and re-creating meshes per frame.** Cache `shared_ptr<mesh>` results from `AssetManager`. Loads are async but still expensive.
